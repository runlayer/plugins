# MCP Server Deployment to Runlayer

Guide for deploying MCP servers to Runlayer using the `uvx runlayer deploy` CLI.

---

## Prerequisites

1. **Runlayer CLI** -- `uvx runlayer`
2. **Authenticated** -- Run `uvx runlayer login` (stores credentials locally so no flags needed on subsequent commands)
3. **Docker** -- Must be installed and running locally (used to build the image)
4. **`runlayer.yaml`** -- Deployment config in project root (generated by `deploy init`)
5. **`Dockerfile`** -- Container definition for the MCP server

---

## Quick Start

```bash
# 1. Login (one-time — stores credentials locally)
uvx runlayer login

# 2. Initialize — creates deployment record and generates runlayer.yaml with ID
uvx runlayer deploy init

# 3. Edit runlayer.yaml — set service.port to match your server

# 4. Deploy
uvx runlayer deploy
```

**Important:** `deploy init` must run first to generate the deployment ID in `runlayer.yaml`. Running `deploy` without an `id` field will fail.

## runlayer.yaml Configuration

Minimal config for an MCP server:

```yaml
name: my-service-mcp-server
build:
  dockerfile: Dockerfile
service:
  port: 3000
  health_check:
    path: /health
    interval: 30
```

Full reference:

```yaml
name: my-service-mcp-server        # Required. Lowercase, URL-friendly.
id: null                            # Auto-populated after first deploy.

build:
  dockerfile: Dockerfile            # Path to Dockerfile
  context: .                        # Build context (default: .)

service:
  port: 3000                        # Port the server listens on
  health_check:
    path: /health                   # Health check endpoint
    interval: 30                    # Seconds between checks
  env:                              # Environment variables
    API_KEY: ${API_KEY}             # Supports ${VAR} substitution from .env file
```

### Environment Variable Substitution

Use `${VAR_NAME}` syntax in `runlayer.yaml` to reference env vars. Pass a `.env` file at deploy time:

```bash
uvx runlayer deploy --env-file .env
```

## Dockerfile Templates

### TypeScript MCP Server

```dockerfile
FROM node:22-slim AS builder
WORKDIR /app
COPY package*.json ./
RUN npm ci --production=false
COPY . .
RUN npm run build

FROM node:22-slim
WORKDIR /app
COPY --from=builder /app/dist ./dist
COPY --from=builder /app/node_modules ./node_modules
COPY --from=builder /app/package.json ./
ENV NODE_ENV=production
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s CMD wget -qO- http://localhost:3000/health || exit 1
CMD ["node", "dist/index.js"]
```

### Python MCP Server

```dockerfile
FROM python:3.13-slim AS builder
WORKDIR /app
COPY pyproject.toml uv.lock ./
RUN pip install uv && uv sync --frozen --no-dev

FROM python:3.13-slim
WORKDIR /app
COPY --from=builder /app/.venv /app/.venv
COPY . .
ENV PATH="/app/.venv/bin:$PATH"
EXPOSE 3000
HEALTHCHECK --interval=30s --timeout=3s CMD python -c "import urllib.request; urllib.request.urlopen('http://localhost:3000/health')" || exit 1
CMD ["python", "-m", "server"]
```

## Deploy Workflow

The CLI performs these steps:

1. **Load config** -- Reads `runlayer.yaml`
2. **Create/update deployment** -- Registers with Runlayer backend
3. **Build Docker image** -- Runs `docker build` locally
4. **Push image** -- Uploads to Runlayer's container registry (ECR)
5. **Trigger deployment** -- Backend provisions infrastructure and starts the service

### CLI Flags

After `runlayer login`, credentials are stored locally. Flags are optional overrides:

| Flag | Short | Description |
|---|---|---|
| `--config` | `-c` | Path to `runlayer.yaml` (default: `runlayer.yaml`) |
| `--secret` | `-s` | API secret (override; not needed after `login`) |
| `--host` | `-H` | Runlayer host URL (override; not needed after `login`) |
| `--env-file` | `-e` | Path to `.env` file for variable substitution |

## Troubleshooting

| Issue | Fix |
|---|---|
| "Configuration file not found" | Run `uvx runlayer deploy init` or create `runlayer.yaml` |
| "API secret required" | Run `uvx runlayer login` or pass `--secret` |
| Docker build fails | Test `docker build .` locally first, fix Dockerfile |
| Push fails (auth) | Verify API key has deploy permissions |
| Health check fails after deploy | Ensure server exposes `/health` endpoint on correct port |
| Env var not substituted | Check `${VAR}` syntax in yaml; pass `--env-file` flag |

## Transport Considerations

Deployed containers run on ECS Fargate. The Runlayer proxy currently connects to them
over the network (AWS Service Discovery DNS) using Streamable HTTP transport.

The deployed container needs to listen on a port. Two approaches:

### Option A: Native HTTP (recommended)

Build the server with Streamable HTTP transport directly:

- Expose a `/mcp` POST endpoint (or configure `service.path` in `runlayer.yaml`)
- Add a `/health` GET endpoint for ECS health checks
- Make the server stateless for horizontal scaling

### Option B: stdio with transport bridge

If the server only supports stdio, wrap it with a transport bridge in the Dockerfile
(e.g. `supergateway` or `mcp-remote`):

```dockerfile
# Example: wrap a stdio server with supergateway
RUN npm install -g supergateway
CMD ["supergateway", "--stdio", "node", "dist/index.js", "--port", "3000"]
```

This exposes the stdio server as HTTP without modifying the server code.

### Upgrading stdio to dual-mode

To add HTTP support alongside stdio:

1. Add a `/health` GET endpoint returning `{"status":"ok"}`
2. Add a `/mcp` POST endpoint using `StreamableHTTPServerTransport`
3. Use `PORT` env var to switch: stdio when unset, HTTP when set
4. Verify locally:
   ```bash
   PORT=3000 node dist/index.js &
   curl http://localhost:3000/health
   ```

### Best practices for deployed servers

- Expose `/health` for ECS health checks
- Make the server **stateless** for horizontal scaling
- Use **OAuth 2.1** or API key auth for security
